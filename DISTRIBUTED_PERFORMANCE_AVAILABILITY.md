## 一、分布式

### 1.1 理论 & 算法 & 协议

#### CAP 理论

| 属性 | 说明 |
|---|---|
| C（Consistency） | 强一致性：所有节点同一时刻看到相同数据 |
| A（Availability） | 可用性：每个请求都收到非错误响应 |
| P（Partition tolerance） | 分区容错性：网络分区时系统仍运作 |

**结论**：P 不可避免，实际是 CP vs AP 的取舍。

| 选择 | 场景 | 典型 |
|---|---|---|
| CP | 强一致优先 | ZooKeeper、etcd |
| AP | 可用性优先 | Eureka、Redis Cluster |

#### BASE 理论

| 属性 | 说明 |
|---|---|
| BA（Basically Available） | 基本可用：允许降级 |
| S（Soft State） | 软状态：允许中间状态 |
| E（Eventually Consistent） | 最终一致：保证最终达到一致 |

**落地**：电商下单→扣库存→加积分，积分可延迟到账（最终一致）。

#### 共识算法

| 算法 | 特点 | 适用 |
|---|---|---|
| Paxos | 理论基础，难理解 | 理论参考 |
| Raft | 易理解，leader-follower | etcd、Consul |
| ZAB | ZK 专用，原子广播 | ZooKeeper |
| Gossip | 最终一致，节点间传播 | Redis Cluster、Cassandra |

#### 一致性哈希

```
问题: 节点增减时 hash%n 导致大量数据迁移
方案: 哈希环 + 虚拟节点

数据 → hash → 定位到环上 → 顺时针找到第一个节点
节点增删只影响相邻区间

虚拟节点: 每个物理节点映射 N 个虚拟节点，解决数据倾斜
```

**脚手架对应**：framework-tools `SnowflakeIdGenerator`（分布式ID）、framework-cache（一致性哈希路由）

### 1.2 RPC

#### RPC 核心模型

```
调用方                         服务方
  ├─ 动态代理(拦截方法调用)       │
  ├─ 序列化(对象→字节) ────────→│
  ├─ 网络通信(Netty NIO) ──────→│
  ├─ 服务发现(注册中心) ────────→│
  ├─ 负载均衡(选节点) ──────────→│
  │                             ├─ 反序列化
  │                             ├─ 反射调用
  │                             ├─ 返回结果
  │←── 原路返回 ────────────────│
```

#### RPC vs HTTP

| 对比 | RPC | HTTP |
|---|---|---|
| 协议 | 自定义二进制 | 文本/二进制 |
| 性能 | 高 | 中 |
| 序列化 | Protobuf/Hessian | JSON |
| 服务治理 | 内置 | 需外部 |
| 跨语言 | gRPC好，Dubbo差 | 好 |
| 适用 | 内部微服务 | 对外API |

#### Dubbo 核心概念

| 概念 | 说明 |
|---|---|
| Provider | 暴露服务的服务提供方 |
| Consumer | 调用远程服务的消费方 |
| Registry | 服务注册与发现的注册中心 |
| Monitor | 统计服务和调用次数 |
| Container | 服务运行容器 |

### 1.4 分布式 ID

#### 方案对比

| 方案 | 唯一性 | 递增 | 性能 | 依赖 |
|---|---|---|---|---|
| UUID | 全局唯一 | 无序 | 高 | 无 |
| DB自增 | 全局唯一 | 递增 | 低 | DB |
| **雪花算法** | 全局唯一 | 趋势递增 | 极高 | 时钟 |
| **号段模式** | 全局唯一 | 趋势递增 | 高 | DB(弱) |
| Redis INCR | 全局唯一 | 递增 | 中 | Redis |

#### 雪花算法结构

```
| 1bit符号位 | 41bit时间戳 | 10bit机器ID | 12bit序列号 |
```

- 41bit 时间戳：可用约69年
- 10bit 机器ID：最多1024台机器
- 12bit 序列号：每毫秒最多4096个ID

**时钟回拨问题**：记录最后时间戳，回拨时自旋等待或抛异常走备用发号器。

**脚手架已实现**：framework-tools `SnowflakeIdGenerator`

### 1.5 分布式锁

#### 三种实现对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| DB唯一索引 | 简单 | 性能差，无续期 |
| Redis SETNX | 高性能 | 主从切换可能丢锁 |
| **Redisson** | 功能完整，看门狗续期 | 依赖Redis |
| ZooKeeper | 强一致 | 性能差，复杂 |

#### Redisson 看门狗机制

```
获取锁(不设leaseTime)
  → 默认30s过期
  → 启动看门狗线程，每10s检查
    → 如果还持有锁 → 续期到30s
    → 如果业务完成释放锁 → 停止看门狗
```

**脚手架已实现**：framework-lock `@DistributedLock`，leaseTime=-1 自动启用看门狗

## 二、高性能

### 2.1 数据库优化

#### 读写分离

```
写 → Master ──复制──→ Slave1
                    ──复制──→ Slave2
                    ──复制──→ Slave3

读 ← Slave1 / Slave2 / Slave3
```

| 问题 | 方案 |
|---|---|
| 主从延迟 | 写后强制读主、半同步复制 |
| 从库负载 | 多从库 + 负载均衡 |

#### 分库分表

| 维度 | 说明 |
|---|---|
| 垂直分库 | 按业务拆库（用户库/订单库/商品库） |
| 垂直分表 | 按字段拆表（热字段+冷字段） |
| 水平分库 | 同一表数据分散到多个库 |
| 水平分表 | 同一库中表数据分散到多张表 |

**分片策略**：

| 策略 | 分片键 | 适用 |
|---|---|---|
| 取模 | user_id % N | 用户相关 |
| 范围 | create_time | 日志/流水 |
| 一致性哈希 | hash(key) | 均匀分布 |
| 租户 | tenant_id | SaaS |

**脚手架对应**：framework-tools `SnowflakeIdGenerator`（分库分表后生成分布式ID）

#### 数据冷热分离

| 策略 | 说明 |
|---|---|
| 冷数据 | 历史订单、归档日志 → 存储到低成本存储（HBase/对象存储） |
| 热数据 | 近期数据 → MySQL/Redis |
| 分界线 | 通常按时间（如3个月前为冷） |

#### SQL 优化

| 手段 | 说明 |
|---|---|
| 索引优化 | 遵循最左匹配、避免索引失效 |
| 避免 SELECT * | 只查需要的字段，利用覆盖索引 |
| LIMIT 优化 | 深度分页用游标 `WHERE id > last_id LIMIT 10` |
| JOIN 优化 | 小表驱动大表，被驱动表有索引 |
| 子查询优化 | 改写为 JOIN |
| 批量操作 | INSERT 批量 > 循环单条 |

#### 深度分页优化

```sql
-- 慢: OFFSET 越大越慢
SELECT * FROM order ORDER BY id LIMIT 1000000, 10;

-- 快: 游标分页
SELECT * FROM order WHERE id > 1000000 ORDER BY id LIMIT 10;

-- 快: 延迟关联
SELECT * FROM order o
INNER JOIN (SELECT id FROM order ORDER BY id LIMIT 1000000, 10) t
ON o.id = t.id;
```

### 2.3 CDN

| 概念 | 说明 |
|---|---|
| CDN | Content Delivery Network，内容分发网络 |
| 原理 | 将静态资源缓存到离用户最近的边缘节点 |
| 命中 | 用户请求 → 就近边缘节点 → 命中直接返回 |

### 2.4 消息队列

#### MQ 对比

| 维度 | RabbitMQ | RocketMQ | Kafka |
|---|---|---|---|
| 语言 | Erlang | Java | Scala |
| 吞吐量 | 万级 | 十万级 | 百万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 可靠性 | 高 | 非常高 | 高 |
| 事务消息 | 不支持 | 支持 | 不支持 |
| 延迟队列 | 死信+插件 | 原生支持 | 不支持 |
| 适用 | 企业级 | 电商/金融 | 大数据/日志 |

#### MQ 三大核心场景

| 场景 | 说明 |
|---|---|
| 异步解耦 | 下单→扣库存→加积分，各服务通过MQ解耦 |
| 削峰填谷 | 秒杀请求先入MQ，消费端按DB承受速度消费 |
| 广播通知 | 配置变更/事件通知，多消费者同时收到 |

#### MQ 可靠性保障

| 环节 | 保障 |
|---|---|
| 生产端 | 确认机制(confirm) + 持久化 |
| Broker | 持久化 + 镜像队列/Quorum |
| 消费端 | 手动ACK + 幂等消费 |
| 死信 | 消费失败→死信队列→人工/自动补偿 |

**脚手架已实现**：framework-mq（完整的生产/消费/死信/重试/管理控制台）

#### Disruptor

| 特点 | 说明 |
|---|---|
| 原理 | 环形数组(RingBuffer) + 无锁CAS |
| 性能 | 单线程每秒600万消息 |
| 机制 | 序号协调生产者-消费者，避免伪共享 |
| 适用 | 超高性能内存队列（如日志异步写入） |

### 3.2 限流

#### 限流算法

| 算法 | 原理 | 优点 | 缺点 |

| 固定窗口 | 时间窗口内计数 | 简单 | 临界点突发 |

### 3.3 降级 & 熔断

#### 降级

| 类型 | 说明 | 示例 |
|---|---|---|
| 主动降级 | 大促前关闭非核心功能 | 关闭推荐/评论 |
| 超时降级 | 接口超时返回兜底 | 推荐超时→返回默认列表 |
| 异常降级 | 异常率超阈值降级 | 支付失败率>10%→降级 |
| 限流降级 | 限流后返回排队提示 | 秒杀限流→"系统繁忙" |

#### 熔断

```
状态机:
  CLOSED → 失败率>50% → OPEN（拒绝请求）
  OPEN → 等待30s → HALF_OPEN（放行10个探测请求）
  HALF_OPEN → 探测成功 → CLOSED（恢复）
  HALF_OPEN → 探测失败 → OPEN（继续熔断）
```

| 参数 | 默认值 | 说明 |
|---|---|---|
| 失败率阈值 | 50% | 触发熔断 |
| 慢调用阈值 | 2s | 慢于此时长计为慢调用 |
| 慢调用比例 | 80% | 超此值触发熔断 |
| 熔断时长 | 30s | OPEN持续时间 |
| 半开请求数 | 10 | HALF_OPEN放行数 |
| 滑窗大小 | 100次 | 统计窗口 |

**脚手架已实现**：framework-retry `@CircuitBreaker`，基于 Resilience4j

### 3.4 超时 & 重试

#### 超时层次

| 层 | 超时设置 | 说明 |
|---|---|---|
| 网关 | 30s | 最外层，兜底 |
| RPC/HTTP | 3-5s | 服务间调用 |
| DB | 1-3s | SQL执行 |
| Redis | 200ms | 缓存操作 |
| MQ | 5s | 消息发送 |

**原则**：上游超时 > 下游超时，避免上游先超时但下游还在执行。

#### 重试策略

| 策略 | 说明 |
|---|---|
| 固定间隔 | 每次等待固定时间 |
| 指数退避 | 间隔递增：1s→2s→4s→8s |
| 随机退避 | 随机间隔，避免惊群 |
| 带上限的指数退避 | 指数增长但不超过上限 |

**重试注意**：

| 问题 | 方案 |
|---|---|
| 重复执行 | 接口幂等 |
| 重试风暴 | 限制最大重试次数 |
| 惊群效应 | 随机退避 |
| 死循环 | 最大重试+熔断 |

**脚手架已实现**：framework-retry `@Retry`，支持固定/指数退避+异常过滤+fallback

### 3.6 灾备 & 异地多活

#### 灾备 = 容灾 + 备份

| 概念 | 说明 |
|---|---|
| 备份 | 数据多份副本（冷备/热备） |
| 容灾 | 异地建立相同系统，故障时切换 |

#### 灾备等级

| 等级 | RTO | RPO | 说明 |
|---|---|---|---|
| 同城双活 | <1min | 0 | 同城两机房互为主备 |
| 两地三中心 | <5min | 秒级 | 同城双活+异地灾备 |
| 异地多活 | <30s | 0 | 多地域同时服务 |

#### 异地多活

```
用户 → 路由层(按userId路由) → 单元A(北京)
                            → 单元B(上海)
                            → 单元C(深圳)

每个单元:
  - 完整服务实例
  - 独立数据库（按用户分片）
  - 独立缓存
  - 跨单元数据同步（DTS/OTTER）
```

**与灾备区别**：异地多活所有站点同时服务，灾备有主备之分，备库不服务。
